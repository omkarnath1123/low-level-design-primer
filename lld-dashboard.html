Understanding SOLID Design Principles

Coined by Robert C. Martin (Uncle Bob), SOLID is an acronym representing five fundamental design principles that ensure system structures are understandable, flexible, and maintainable over time.

S

Single Responsibility Principle (SRP)

"A class should have only one reason to change"

A class must focus on a single job, responsibility, or business logic sector. Mixing email dispatch, db transactions, and report formatting inside a single module violates SRP.

Common Fix:

Extract logging, database management, and formatting into separate helper/service classes.
O

Open/Closed Principle (OCP)

"Software entities should be open for extension, but closed for modification"

You should be able to extend class behavior without modifying its existing code base, which prevents regression bugs in previously tested logic.

Common Fix:

Introduce interfaces or abstract classes, then implement strategies (e.g., polymorphism) for new behaviors.
L

Liskov Substitution Principle (LSP)

"Derived classes must be substitutable for their base classes"

Subclasses should extend the behavior of base classes without changing or breaking their original expectations. Throwing `UnsupportedOperationException` in a subclass violates LSP.

Common Fix:

Do not force-inherit. Create more focused interface partitions or use composition over inheritance.
I

Interface Segregation Principle (ISP)

"Clients should not be forced to depend on methods they do not use"

Many client-specific interfaces are better than one general-purpose interface. Don't force implementing classes to define dummy behavior for unneeded methods.

Common Fix:

Break a bloated multi-method interface into several smaller, highly cohesive interfaces.
D

Dependency Inversion Principle (DIP)

"Depend on abstractions, not concretions"

High-level modules should not depend on low-level modules. Both should depend on abstractions (interfaces). Changes in deep helper details should not break core system orchestration.

Common Fix:

Use Dependency Injection (DI) frameworks or parameterize constructors with interface specifications.

Creational Design Patterns

Creational patterns handle object creation mechanisms, optimizing how objects are instantiated. They encapsulate instantiation details, decoupling the system from concrete constructors.

Creational

Singleton Pattern

Ensures a class has only one instance and provides a global access point to it. Commonly used for database connections, caching classes, thread pools, and global loggers. 

public class DatabaseConnection {
    private static volatile DatabaseConnection instance;
    private DatabaseConnection() {} // Private Constructor

    public static DatabaseConnection getInstance() {
        if (instance == null) {
            synchronized (DatabaseConnection.class) {
                if (instance == null) {
                    instance = new DatabaseConnection();
                }
            }
        }
        return instance;
    }
}
Creational

Factory Method Pattern

Defines an interface for creating objects, but lets subclasses decide which class to instantiate. Decouples client callers from instantiation logic. 

public interface Notification {
    void notifyUser();
}
public class NotificationFactory {
    public Notification createNotification(String channel) {
        if (channel.equalsIgnoreCase("SMS")) return new SMSNotification();
        if (channel.equalsIgnoreCase("EMAIL")) return new EmailNotification();
        throw new IllegalArgumentException("Unknown channel");
    }
}

Structural Design Patterns

Structural patterns explain how to assemble objects and classes into larger, flexible structures while keeping system boundaries highly modular and clean.

Structural

Adapter Pattern

Allows objects with incompatible interfaces to collaborate. Often used to translate vintage APIs or third-party components into expected interface structures. 

// Converts Legacy XML Stock Service to modern JSON Stock Parser
public class XMLToJSONAdapter implements JSONStockData {
    private LegacyXMLStockService xmlService;
    
    public XMLToJSONAdapter(LegacyXMLStockService xmlService) {
        this.xmlService = xmlService;
    }
    
    @Override
    public JSONPayload getStockPayload() {
        String xml = xmlService.getXMLData();
        return convertXMLToJSON(xml);
    }
}
Structural

Decorator Pattern

Attaches new behaviors to objects dynamically by placing them inside special wrapper classes that contain the behaviors. 

// Wrap basic Coffee objects with dynamic toppings dynamically
public class WhipDecorator extends CoffeeDecorator {
    public WhipDecorator(Coffee coffee) {
        super(coffee);
    }
    @Override
    public double getCost() {
        return super.getCost() + 0.50; // Extra cost of Whip
    }
}
Structural

Composite Pattern

Composes objects into tree structures and lets clients treat individual objects and compositions uniformly. The Parking Lot (Lot → Floor → Spot) is a textbook composite. 

// Lot contains Floors; Floor contains Spots
public class ParkingFloor {
    private List<ParkingSpot> spots;
    public ParkingSpot findAvailable(VehicleType t) {
        return spots.stream()
            .filter(s -> s.isFree() && s.fits(t))
            .findFirst().orElse(null);
    }
}
Structural

Facade Pattern

Provides a simplified, unified interface to a complex subsystem. The Cache facade hides the storage + eviction-policy wiring behind plain get/put. 

public class Cache<K,V> {           // facade
    private final Storage<K,V> storage;
    private final EvictionPolicy<K> policy;
    public V get(K k) { ... }       // hides DLL + hashmap
    public void put(K k, V v) { ... }
}

Behavioral Design Patterns

Behavioral patterns focus on algorithms and the assignment of responsibilities between objects, defining clean communication networks and logical processing chains.

Behavioral

Strategy Pattern

Defines a family of algorithms, encapsulates each one, and makes them interchangeable. Strategy lets the algorithm vary independently from clients that use it. 

public interface EvictionStrategy {
    void evict(Cache cache);
}
public class LRUEviction implements EvictionStrategy {
    public void evict(Cache cache) {
        // Evicts Least Recently Used element from the Cache
    }
}
Behavioral

Observer Pattern

Defines a subscription mechanism to notify multiple objects about any events that happen to the object they're observing. 

public interface Observer {
    void update(String status);
}
public class DeliveryAgent implements Observer {
    public void update(String status) {
        System.out.println("Status updated to: " + status);
    }
}
Behavioral

State Pattern

Lets an object alter its behavior when its internal state changes — it appears to change its class. Ideal for a Vending Machine (Idle → HasMoney → Dispensing) or a parking spot's occupancy lifecycle. 

public interface VendingState {
    void insertCoin(VendingMachine m);
    void dispense(VendingMachine m);
}
public class HasMoneyState implements VendingState {
    public void dispense(VendingMachine m) {
        m.releaseProduct();
        m.setState(new IdleState());
    }
}
Behavioral

Command Pattern

Encapsulates a request as an object, letting you parameterise, queue, and undo/redo operations. Powers a TextPad's multi-level undo and home-automation remote buttons. 

public interface Command {
    void execute();
    void undo();
}
public class InsertTextCommand implements Command {
    public void execute() { doc.insert(text); }
    public void undo()    { doc.delete(text); }
}

Case Study 2

Design an In-Memory LRU Cache

A generic, bounded, thread-safe key-value cache with a pluggable eviction policy and swappable storage backend. Achieves O(1) get and O(1) eviction by pairing a doubly-linked list with a hashmap.

  • DLL gives O(1) arbitrary node removal; HashMap gives O(1) lookup.
  • Swap LRU for LFU/FIFO without touching the Cache facade (Strategy).
  • put() evicts then retries on StorageFullException (Template Method).

Core Pattern Checklist

Case Study 4

Distributed ID Generator (Twitter Snowflake)

Generate globally unique, time-sortable 64-bit IDs across many nodes without coordination — 4096 IDs/ms/node × 1024 nodes = ~4M IDs/ms globally.

  • Monotonically increasing & sortable by creation time.
  • Clock-drift protection throws ClockMovedBackException.
  • Thread-safe via synchronized block; node ID validated at startup.

64-bit ID Layout

41 bits timestamp (~69 yrs)
10 bits node id (0–1023)
12 bits sequence (0–4095)

// generate one Snowflake ID (synchronized)

currentTs = now() - EPOCH_START;
if (currentTs < lastTimestamp) throw ClockMovedBackException;
if (currentTs == lastTimestamp) {
    sequence = (sequence + 1) & 4095;        // mod 4096
    if (sequence == 0) waitNextMillis();     // overflow
} else {
    sequence = 0;
}
lastTimestamp = currentTs;
id = (currentTs << 22) | (nodeId << 12) | sequence;
return id;
